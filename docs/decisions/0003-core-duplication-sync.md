# ADR-0003: デスクトップ版とのコア重複の同期戦略

- ステータス: 採用
- 日付: 2026-06-26
- 決定者: 作者（gatowostudio / dekka）／参謀システム（issue #7）
- 関連: ADR-0001（Tauri 流用・2リポ構成）, ADR-0002（状態所有権・計時）

## 背景

本リポは デスクトップ版 [`C:\Dev\simpomo`](https://github.com/gatowostudio/simpomo) からタイマーコア・統計・
音生成・Rust↔TS 同期契約を**コードごと流用**している（ADR-0001）。2 リポジトリに同じコードの実体が在るため、
片方でコアを直しても他方へは自動で伝播しない。`#[serde(default)]` 等のゆるい契約のせいで、ズレても
例外を出さず静かに進行する（例: android だけ新設定が効かない）。リポ間ドリフトを検出する仕組みが無いのが
課題だった。

さらに移植の過程で、当初「バイト一致のコピー」と記していたファイルの一部が**意図的に分岐**した。
2026-06-26 時点の実測（`sha256` 比較）:

| ファイル | 状態 | 分岐理由 |
|---|---|---|
| `src-tauri/src/stats.rs` | **一致** | — |
| `src/lib/sounds.ts` | **一致** | — |
| `src/lib/bgm.ts` | **一致** | — |
| `src/lib/timer.ts` | **一致** | — |
| `src/lib/stats.ts` | **一致** | — |
| `src/lib/audio.ts` | **一致** | — |
| `src/lib/color.ts` | **一致** | — |
| `src-tauri/src/timer.rs` | 分岐（加算的） | android が永続化フック（`TimerState`/`restore`/`restore_for_launch`/`Deserialize`）を追加（ADR-0002 §3） |
| `src-tauri/src/settings.rs` | 分岐（削減的） | desktop 専用フィールド（`corner`/`skipTaskbar`）を削除 |
| `src/lib/settings.ts` | 分岐（削減的） | 同上（`Corner`/`openSettings` 等を削除） |
| `src/lib/notify.ts` | 分岐（置換的） | 前景判定を `isVisible()`→`document.visibilityState` に置換（Android 対応） |

つまり「全部バイト一致」は既に事実と異なる。同期戦略は、この2クラスを**区別して**扱う必要がある。

## 決定

共有ファイルを 2 クラスに分類し、扱いを分ける。**どのファイルがどのクラスか（membership）の正本は
`scripts/check-core-drift.ps1` の `$mirror` / `$diverged`**（実際に enforce するのはスクリプトなので）。
上の表はその根拠（一致/分岐の理由）を人間向けに併記したもので、スクリプトと一致させる。以下はクラスごとの
扱いだけを述べ、ファイル名は再列挙しない（再列挙は二重管理＝ドリフト源になるため。具体名は上表/スクリプト参照）。

### クラス A: ミラー（content-identical 契約を保つ）

プラットフォーム非依存の純ロジック。**両リポで一致を維持する**。どちらかを直したら、もう片方へ
**そのままコピー**する。`check-core-drift.ps1` がこの一致を検査し（改行は LF 正規化して比較）、
ズレていれば失敗（exit 1）させる。

### クラス B: 分岐（共有の系譜・差分は意図的）

プラットフォーム差のため**意図的に分岐**している。一致は求めない。ただし**共有している中核ロジック**
（`timer.rs` の状態機械＝tick/遷移、設定のキー名・サニタイズ規則、Rust↔TS の camelCase 契約）は両リポで
一致させ続ける。`check-core-drift.ps1` はこのクラスを「差分あり」と分かった上で**情報表示のみ**
（失敗させない）にし、変更時の手動照合を促す。

- `timer.rs` の分岐は**加算的**: android 固有の追加は ADR-0002 §3 由来であり、状態機械本体には触れていない。
  状態機械を直すときは両リポに反映する。
- `settings.*` の分岐は**削減的**: android は desktop のスーパーセットから列を削っただけ。新設定を足すときは
  両リポの `*.rs` と `*.ts` の双方（4 ファイル）に同じキーを追加する。

## 運用ルール（コアを直すときの手順）

1. コア（状態機械 / 統計 / 音生成 / 設定キー / Rust↔TS 契約）を変更したら、**両リポに反映**する。
2. push 前にローカルで `pwsh scripts/check-core-drift.ps1` を実行し、クラス A のドリフトが無いことを確認する。
   クラス B は差分サマリを目視し、中核ロジックの取りこぼしが無いか確認する。手動実行忘れを避けるため
   `git config core.hooksPath scripts/hooks` を一度設定すると、`pre-push` フックが push のたびに検査する。
3. 分類が変わったら（ミラーが分岐した / 分岐が解消した）、**まず正本 `check-core-drift.ps1` を直し**、
   次に本 ADR 冒頭の表を合わせる（この 2 つだけが membership を持つ。`CLAUDE.md` 等は再列挙せず ADR/スクリプトを
   指すだけにしてある＝直す箇所を 2 つに抑える）。

## 検討した代替案

- **共有 crate / npm パッケージ化**: 重複の根治。ただし 2 リポ・1 個人保守の現状ではセットアップ/公開の
  オーバーヘッドが利得を上回る（ADR-0001 が 2 リポ別管理を選んだのと同じ理由）。
- **desktop リポへ Android ターゲット統合（1 リポ化）**: 重複を物理的に消すが、デスクトップ版の CI/リリース・
  依存構成を Android で汚す。現状は分離の利を取る。
- 上記いずれも将来の見直しトリガーは「コア変更が頻発し、手動同期が破綻し始めたとき」。それまでは
  **検出スクリプト＋手順**という軽量策で十分とする。

## 限界

- `check-core-drift.ps1` は**ローカルに両リポが在る前提**（既定 `C:\Dev\simpomo`）。CI には desktop リポが
  無いので走らせない（CI で守るものではなく、保守者のローカル前ゲート）。
- **強制は opt-in**: 検査は手動実行か、`core.hooksPath` を設定して初めて pre-push で走る。設定しなければ
  「同期を忘れる人は検査も忘れる」失敗相関が残る（沈黙ドリフトを完全には防げない）。これは承知の上で、
  軽量策として割り切る。確実な強制が要るようになったら共有 crate 化（根治）を前倒しする。
- クラス B の「中核ロジック一致」は機械判定できない（差分の意味づけが要る）。スクリプトは差分の存在を示すだけで、
  正しさの最終判断は人間が行う。
- 比較は改行を LF 正規化してから行う（`.gitattributes`/`autocrlf` の差で実質同一が DRIFT 誤検出するのを防ぐ）。
